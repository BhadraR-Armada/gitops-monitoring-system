# ArgoCD Monitoring & Alerting Stack

An automated deployment system for monitoring Kubernetes applications managed by ArgoCD, featuring Prometheus metrics, Grafana dashboards, and Microsoft Teams alerting.

## Overview

This project sets up a full observability stack on a Kubernetes cluster:

- **ArgoCD** — GitOps continuous delivery tool
- **Prometheus** — Metrics collection and alerting
- **Grafana** — Visualisation dashboards for ArgoCD application health and sync status
- **Alertmanager** — Alert routing and deduplication
- **prometheus-msteams** — Proxy that forwards Prometheus alerts to Microsoft Teams

### Alert Flow

```
Prometheus → Alertmanager → prometheus-msteams proxy → Microsoft Teams
```

- A **Microsoft Teams** channel with an **Incoming Webhook** URL configured (see [Teams Setup](#microsoft-teams-setup) below)

## Microsoft Teams Setup

Before running the install script, you need to create an Incoming Webhook in your Teams channel:

1. Go to the Teams channel where you want to receive alerts
2. Click `···` next to the channel name → **Manage channel**
3. Click **Edit** next to Connectors
4. Search for **Incoming Webhook** → **Add** → **Configure**
5. Give it a name (e.g. `Alertmanager`) and optionally upload an image
6. Click **Create** and copy the webhook URL
7. Click **Done**

Keep this URL handy — the install script will prompt you for it.

## Installation

### 1. Clone the repository

```bash
git clone https://github.com/BhadraR-Armada/gitops-monitoring-system.git
cd gitops-monitoring-system
```

### 2. Make the script executable

```bash
chmod +x scripts/install.sh
```

### 3. Run the install script

```bash
./scripts/install.sh
```

The script will prompt you to enter your Microsoft Teams webhook URL if it is not already set as an environment variable.

Alternatively, you can pass it directly:

```bash
TEAMS_WEBHOOK_URL="https://your-org.webhook.office.com/..." ./scripts/install.sh
```

### 4. Access the UIs

Once the script completes, run each of these commands in a separate terminal to access the UIs:

```bash
# ArgoCD
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Prometheus
kubectl port-forward pod/prometheus-monitoring-kube-prometheus-prometheus-0 -n monitoring 9090:9090

# Grafana
kubectl port-forward -n monitoring \
  $(kubectl get pod -n monitoring -l "app.kubernetes.io/name=grafana,app.kubernetes.io/instance=monitoring" -o name) 3000

# Alertmanager
kubectl port-forward svc/alertmanager-operated -n monitoring 9093:9093
```

Then open in your browser:

| Service | URL | Default Credentials |
|---|---|---|
| ArgoCD | https://localhost:8080 | admin / (printed by script) |
| Grafana | http://localhost:3000 | admin / (printed by script) |
| Prometheus | http://localhost:9090 | none |
| Alertmanager | http://localhost:9093 | none |

## Testing the Alerting System

To verify the full alerting pipeline is working, you can deliberately break and fix the sample todo application and observe alerts firing in both Alertmanager and your Teams channel.

### Breaking the Application

Ensure that auto-healing and pruning is not enabled. Apply a memory limit that is too low for the application to run, causing it to crash loop:
```bash
kubectl patch deployment python-app-todo -n todoapp --type='json' \
  -p='[{"op": "add", "path": "/spec/template/spec/containers/0/resources", "value": {"limits": {"memory": "1Mi"}}}]'
```

After a few minutes you should see:
- The pod entering `CrashLoopBackOff` state in ArgoCD
- `ApplicationDegraded` and `ApplicationOutOfSync` alerts firing in Alertmanager at `http://localhost:9093`
- Alert notifications appearing in your Microsoft Teams channel

You can watch the pod status with:
```bash
kubectl get pods -n todoapp -w
```

### Fixing the Application

Remove the memory limit to restore the application to a healthy state:
```bash
kubectl patch deployment python-app-todo -n todoapp --type='json' \
  -p='[{"op": "remove", "path": "/spec/template/spec/containers/0/resources"}]'
```

After a few minutes you should see:
- The pod returning to `Running` state
- Alerts resolving in Alertmanager
- A resolved notification appearing in your Teams channel (if `send_resolved: true` is set)

## Customisation

### Changing the Teams Webhook URL

Update the `prometheus-msteams` Helm release with your new URL:

```bash
helm upgrade prometheus-msteams prometheus-msteams/prometheus-msteams \
  --namespace monitoring \
  --set connectors[0].alertmanager="https://YOUR_NEW_WEBHOOK_URL"
```

### Changing Which Alerts Are Sent to Teams

Edit the `routes` section in `monitoring/values.yaml`. By default only ArgoCD-related alerts are forwarded:

```yaml
alertmanager:
  config:
    route:
      routes:
        - matchers:
            - alertname =~ "Application.*"  # matches ApplicationOutOfSync, ApplicationDegraded etc.
          receiver: "teams"
        - matchers:
            - namespace = "argocd"          # matches any alert from the argocd namespace
          receiver: "teams"
```

To send all alerts regardless of namespace or name, remove the routes and set the default receiver to `teams`:

```yaml
route:
  receiver: "teams"  # send everything to Teams
```

Then apply the change:

```bash
helm upgrade monitoring prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --reuse-values \
  --values monitoring/values.yaml
```

### Changing Alert Rules

Edit `manifests/prometheus-rules.yaml` and reapply:

```bash
kubectl apply -f manifests/prometheus-rules.yaml
```

### Changing How Often Alerts Are Repeated

Edit the `repeat_interval` in `monitoring/values.yaml`:

```yaml
alertmanager:
  config:
    route:
      repeat_interval: 6h   # change to your preferred interval
```

Then apply via Helm as shown above.

### Changing the Grafana Dashboard

Edit `grafana/dashboard.json` and rerun the dashboard section of the script:

```bash
kubectl create configmap argocd-status-dashboard \
  --namespace monitoring \
  --from-file=argocd-status.json=./grafana/dashboard.json \
  --dry-run=client -o yaml > manifests/dashboard-configmap.yaml

sed -i '/metadata:/a \  labels:\n    grafana_dashboard: "1"' manifests/dashboard-configmap.yaml
kubectl apply -f manifests/dashboard-configmap.yaml
```

### Pointing ArgoCD at a Different Git Repository

Edit `manifests/argocd-application.yaml` and update the `repoURL` and `path` fields, then reapply:

```bash
kubectl apply -f manifests/argocd-application.yaml -n argocd
```

## Troubleshooting

### Alerts not appearing in Teams

Check the Alertmanager logs:
```bash
kubectl logs alertmanager-monitoring-kube-prometheus-alertmanager-0 \
  -n monitoring -c alertmanager --since=10m
```

Check the prometheus-msteams proxy logs:
```bash
kubectl logs -n monitoring -l app=prometheus-msteams --since=10m
```

Test the Teams webhook URL directly:
```bash
curl -X POST -H "Content-Type: application/json" \
  -d '{"text":"Test alert"}' \
  "https://YOUR_WEBHOOK_URL"
```

### Grafana dashboard not showing data

Ensure the ServiceMonitor is correctly labelled:
```bash
kubectl label svc argocd-metrics -n argocd release=monitoring
```

### ArgoCD applications not syncing

Check ArgoCD application status:
```bash
kubectl get applications -n argocd
```

### Pods not starting

Check pod status and events:
```bash
kubectl get pods -n monitoring
kubectl describe pod <pod-name> -n monitoring
```
